# Data Dictionary (черновик)

Этот документ описывает ключевые поля датасетов `income_train` и `income_test`.  
Основано на EDA и файлах `features_description.csv` (при наличии).

## Общая информация

- `income_train.csv`
  - Строк: 76 786
  - Колонок: 224
  - Содержит `target` (доход) и `w` (вес наблюдения).
- `income_test.csv`
  - Строк: 73 214
  - Колонок: 222
  - Структура фич совпадает с train, **кроме** отсутствия `target` и `w`.

- Период по полю `dt`: с `2024-01-31` по `2024-06-30`.

## Ключевые поля

### Идентификаторы и время

- **`id`**
  - Тип: `int64`
  - Описание: уникальный идентификатор клиента/заявки.
  - Пропуски: нет.

- **`dt`**
  - Тип: `datetime`
  - Описание: дата среза / расчёта признаков.
  - Пропуски: нет.
  - Диапазон: `2024-01-31` – `2024-06-30`.

### Целевое и веса (только train)

- **`target`**
  - Тип: `float64`
  - Описание: целевой показатель — оценка дохода клиента.
  - Пропуски: нет.
  - Статистика:
    - count: 76 786
    - mean: ≈ 92 648
    - std: ≈ 112 409
    - min: 20 000
    - 25%: ≈ 39 710
    - 50% (median): ≈ 62 754
    - 75%: ≈ 100 202
    - max: 1 500 000
  - Комментарий: распределение правостороннее, тяжёлый хвост.

- **`w`**
  - Тип: `float64`
  - Описание: вес наблюдения для WMAE.
  - Пропуски: нет.
  - Статистика:
    - min: ≈ 0.000021
    - max: ≈ 2.570703
    - mean: ≈ 0.569
    - медиана: ≈ 0.449
    - нулевых значений: 0.
  - Комментарий: веса варьируются, что важно учитывать при обучении и оценке модели.

## Социо-демографические поля

- **`gender`**
  - Тип (сырые данные): `object`
  - Интерпретация: категориальный (низкая кардинальность).
  - Значения: `"Женский"`, `"Мужской"`.
  - Пропуски: низкие/отсутствуют (по результатам EDA).

- **`adminarea`**
  - Тип: `object`
  - Интерпретация: категориальный (регион, средняя кардинальность).
  - Кол-во уникальных: 85.
  - Примеры: `Свердловская область`, `Краснодарский край`, `Новосибирская область` и т.д.

- **`city_smart_name`**
  - Тип: `object`
  - Интерпретация: категориальный (город, средняя кардинальность).
  - Кол-во уникальных: 105.
  - Примеры: `Екатеринбург`, `Краснодар`, `Томск`, `Новосибирск`, `Москва` и т.д.

## BKI-поля (лимиты, задолженности)

Примеры колонок (все хранились как `object`, но по факту числовые):

- **`hdb_bki_total_max_limit`**
  - Тип (сырые данные): `object` (строка вида `"52800.0"`)
  - Интерпретация: `numeric_from_string`
  - Описание: общий максимальный лимит по всем продуктам в BKI.
  - Кол-во уникальных: 26 616.

- **`hdb_bki_total_cc_max_limit`**
  - Тип: `object` → `numeric_from_string`
  - Описание: максимальный лимит по кредитным картам.
  - Кол-во уникальных: 6 385.

- **`hdb_bki_total_pil_max_limit`**
  - Тип: `object` → `numeric_from_string`

- **`hdb_bki_active_cc_max_limit`**
  - Тип: `object` → `numeric_from_string`

- **`hdb_bki_other_active_pil_outstanding`**
  - Тип: `object` → `numeric_from_string`
  - Примеры значений: `124714.828125`, `267319.0`, `587252.3125`, `30000.0`, `3634028.763671875`.

- **`hdb_bki_total_products`**
  - Тип: `object` → `numeric_from_string`
  - Примеры: `"6.0"`, `"5.0"`, `"8.0"`, `"0.0"`, `"14.0"` и т.д.
  - Описание: общее количество продуктов по BKI.

_(Аналогично описываются остальные BKI-поля — лимиты, задолженности, просрочки.)_

## Доход/зарплата и связанные показатели

- **`salary_6to12m_avg`**
  - Тип: `float64`
  - Описание: средний доход по зарплатным поступлениям за 6–12 месяцев.
  - Пропуски: есть, но не критичные (точные доли см. в CSV-репортах).

- **`first_salary_income`**
  - Тип: `float64`
  - Описание: первый зафиксированный доход по зарплатному проекту.
  - Пропуски: около 88%.
  - Комментарий: фича-кандидат на утечку (может дублировать/сильно коррелировать с таргетом).

- **`dp_payoutincomedata_payout_*`**
  - Тип: `float64`
  - Описание: статистика по выплатам/доходам (avg/max/sum за периоды 3/6/12 месяцев, предыдущий год).
  - Пропуски: 85–90%.
  - Комментарий: сильные кандидаты на утечку (содержат информацию о доходах, измеренных близко к целевому).

## Транзакционные и поведенческие фичи

Примеры:

- **`turn_cur_cr_avg_act_v2`**
  - Тип: `float64`
  - Описание: средний оборот по текущему кредиту (активная фаза).

- **`transaction_category_supermarket_sum_cnt_d15`**
  - Тип: `float64`
  - Описание: количество/сумма транзакций по категории супермаркеты за последние 15 дней.

- **`loan_cur_amt`**
  - Тип: `float64`
  - Описание: текущий объём кредитной задолженности клиента.

- **`avg_balance_rur_amt_1m_af`**
  - Тип: `float64`
  - Описание: средний остаток по рублёвым счетам за последний месяц.

_(Полный список см. в отчётах `eda_basic_summary_train.csv` и `data_quality_report.csv`.)_

## Пропуски

- Колонок **без пропусков**: 7.
- Много признаков с высокой долей пропусков (> 80–90%), в основном:
  - детализированные категории трат,
  - некоторые BKI-поля,
  - payout/доходные показатели.

Полный список пропусков по колонкам — в файле `ml/reports/missing_share_train.csv`.

```markdown
# EDA Summary (income_train / income_test)

## 1. Структура данных

- **Train**
  - 76 786 наблюдений
  - 224 колонки
  - Содержит:
    - `target` — целевой доход
    - `w` — вес наблюдения для WMAE

- **Test**
  - 73 214 наблюдений
  - 222 колонки
  - Структура фич совпадает с train, **кроме** отсутствия `target` и `w`.

- Период по полю `dt`: с `2024-01-31` по `2024-06-30`.

## 2. Целевой показатель (`target`)

- Тип: `float64`
- Диапазон:
  - min: 20 000
  - max: 1 500 000
- Распределение:
  - mean: ≈ 92 648
  - median: ≈ 62 754
  - 25%: ≈ 39 711
  - 75%: ≈ 100 202
- Вывод:
  - Распределение дохода сильно скошено вправо (тяжёлый хвост).
  - Медиана заметно ниже среднего → много клиентов с высокими доходами.

## 3. Веса (`w`)

- Тип: `float64`
- Диапазон:
  - min: ≈ 0.000021
  - max: ≈ 2.570703
- Описание:
  - mean: ≈ 0.569
  - median: ≈ 0.449
  - нулевых значений нет.
- Вывод:
  - Веса неоднородны, что важно:
    - при обучении модель должна учитывать `w` как `sample_weight`,
    - при оценке качества ключевая метрика — WMAE.

## 4. Временная структура (`dt`)

- Период: январь–июнь 2024.
- Для большинства записей дата присутствует.
- Рекомендация для ML:
  - использовать **time-based CV**:
    - например, train на первых месяцах, валидация на последних (rolling или hold-out на июне),
  - не перемешивать данные по времени бездумно, чтобы не переоценить качество.

## 5. Типы признаков

- В train:
  - числовые (`int`, `float`): основная часть признаков,
  - `object` (строки): 40+ колонок,
  - `datetime`: `dt`.
- Среди `object`:
  - «настоящие» категориальные:
    - `gender` (2 значения),
    - `adminarea` (85 регионов),
    - `city_smart_name` (105 городов),
    - должность `dp_ewb_last_employment_position` (~6000 значений),
    - различные BKI-счётчики (часть из них представлена строками, но по смыслу числовые).
  - числовые, записанные строками:
    - лимиты, задолженности, количество продуктов и т.п. (`hdb_bki_total_max_limit`, `..._max_limit`, `..._outstanding`, и др.).

Вывод: требуется приведение типов:
- `numeric_from_string` → float,
- категориальные → отдельное перечисление (low/high-cardinality).

## 6. Пропуски

- Только **7 колонок без пропусков**.
- Много признаков с очень высокой долей NaN (> 80–90%), в том числе:
  - детализированные категории трат (`avg_by_category__amount__sum__cashflowcategory_name__*`),
  - некоторые депозитные и payout-поля (`dp_payoutincomedata_*`),
  - часть BKI-показателей.

Рекомендация:
- в ML-пайплайне:
  - для числовых — медианная/специальная иммутация + возможный отсев фич с экстремально высокой долей NaN,
  - для категориальных — отдельная категория `"missing"`.

## 8. Выводы для следующего шага (ML)

Для Data Scientist:

1. Использовать предоставленные списки фич (numeric / categorical_low_card / categorical_high_card) из `feature_lists_autodetected.json`.
2. Явно исключить `target` и `w` из фич.
3. В модели:
   - основная метрика: **WMAE** с `w` как весами,
   - схема CV — time-based.
4. Особое внимание уделить:
   - обработке high-cardinality категориальных (должность, addrref и т.п.),
   - проверке влияния payout/доходных фич на риск утечки.
